# Traits

Overview of the core traits that models and payloads use in Dinoco.

---

## What are traits in Dinoco

Traits define the ORM's base behavior: how a model exposes metadata, how it projects columns, how it inserts, how it updates, and how nested payloads are processed.

In codegen-generated models, most of this is already automatically implemented.

## Model Trait

`Model` defines the basic metadata of the model, such as `Include`, `Where`, and the table name.

```rust
impl dinoco::Model for User {
    type Include = UserInclude;
    type Where = UserWhere;

    fn table_name() -> &'static str {
        "users"
    }
}
```

## Projection Trait

`Projection` defines how a struct reads columns from a model and which columns it needs to load.

```rust
impl dinoco::Projection<User> for UserSummary {
    fn columns() -> &'static [&'static str] {
        &["id", "name"]
    }
}
```

It is widely used with `select::&lt;T&gt;()`, `include`, and derived structs with `Extend`.

## InsertModel Trait

`InsertModel` describes how Dinoco should assemble an insert row for the model.

```rust
impl dinoco::InsertModel for User {
    fn insert_columns() -> &'static [&'static str] {
        &["id", "email", "name"]
    }
}
```

In generated models, this trait comes ready-made.

## UpdateModel Trait

`UpdateModel` describes the fields that can be sent in an update and how to identify the record to be changed.

```rust
impl dinoco::UpdateModel for User {
    fn update_columns() -> &'static [&'static str] {
        &["email", "name"]
    }
}
```

## InsertPayload Trait

`InsertPayload` allows `.values(...)` to accept something different from the raw model, separating the payload into:

- base model to be inserted
- nested content to be processed later

This is the trait that `#[insertable]` automatically implements.

## InsertConnectionPayload Trait

`InsertConnectionPayload` is used when the nested payload represents connections to existing records.

It is the fitting used by codegen-generated enums, such as `ArticleConnection`.

```rust
impl dinoco::InsertConnectionPayload<Article> for ArticleConnection {
    fn relation_links(&self, parent: &Article) -> Vec<dinoco::RelationLinkPlan> {
        match self {
            Self::Label(label_id) => vec![dinoco::RelationLinkPlan {
                table_name: "_ArticleLabels",
                columns: &["article_id", "label_id"],
                row: vec![parent.id.clone().into(), label_id.clone().into()],
            }],
        }
    }
}
```

## Traits implemented by the model

In a Dinoco-generated model, it's most common for you to already receive implementations of:

- `Model`
- `Projection`
- `InsertModel`
- `UpdateModel`
- `FindAndUpdateModel`
- `RelationMutationModel`

Depending on the schema's relationships, the model may also receive:

- `InsertRelation&lt;T&gt;`
- `InsertConnection&lt;T&gt;`

## Example with traits and derive together

In practice, the most common flow is:

- `Rowable` for row reading
- `Extend` for projections or enriched payloads
- `#[insertable]` for nested inserts
- traits like `Model`, `Projection`, and `InsertModel` automatically generated by codegen

```rust
#[derive(Debug, Clone, dinoco::Extend)]
#[extend(Article)]
#[insertable]
struct ArticleWithLabels {
    id: String,
    title: String,
    labels: Vec<ArticleConnection>,
}

dinoco::insert_into::<Article>()
    .values(ArticleWithLabels {
        id: "article-30".into(),
        title: "Core Example".into(),
        labels: vec![ArticleConnection::Label("label-10".into())],
    })
    .execute(&client)
    .await?;
```

## When to use each trait

- Use `Model` to identify the table, `Where`, and `Include`.
- Use `Projection` when the struct needs to be loaded by `select`.
- Use `InsertModel` and `UpdateModel` to understand the ORM's write path.
- Use `InsertPayload` when working with nested payloads.
- Use `InsertConnectionPayload` when the nested payload connects existing records.
