# Warteschlangen

Das Dinoco-Warteschlangensystem ermöglicht die Veröffentlichung asynchroner Ereignisse aus `insert`, `find` und `update`, wobei nur die notwendigen Daten in `Redis` gespeichert werden, um den Datensatz neu zu hydrieren, wenn der Worker ausgeführt wird.

## Was Sie tun können

- Ermöglichen Sie das Verketten von .enqueue(...), .enqueue_in(...) und .enqueue_at(...) in allen Client-Methoden, außer bei Löschvorgängen.
- Verarbeiten Sie Jobs mit `workers().on::\<Model\>(... )`.
- Laden Sie Projektionen mit Beziehungen im Worker mithilfe von `workers().on_with_relation::\<Model, Projection\>(... )`.
- Laden Sie die neuesten Daten neu, bevor der Handler ausgeführt wird.
- Ignoriert automatisch Jobs, deren Datensatz nicht mehr existiert.

## Wie es funktioniert

Wenn Sie `enqueue` verwenden, tut Dinoco Folgendes:

1. Führt den Hauptvorgang normal aus.
2. Speichert nur die Suchkriterien des Datensatzes in der Warteschlange.
3. Im Worker werden die neuesten Daten aus der Datenbank abgerufen.
4. Erst dann wird der Handler ausgeführt.

Wenn der Datensatz zum Zeitpunkt der Suche nicht mehr existiert, wird der Handler nicht ausgeführt und der Job wird aus der Warteschlange entfernt.

## Warteschlangenmethoden

- `.enqueue(evento)`: plant die sofortige Ausführung.
- `.enqueue_in(evento, delay_ms)`: plant mit einer Verzögerung in Millisekunden.
- `.enqueue_at(evento, data_utc)`: plant für ein bestimmtes Datum in `chrono::DateTime\<Utc\>`.

```rust
use database::*;

insert_into::<User>()
    .values(User { id: 1, name: "Matheus".to_string() })
    .enqueue("user.created")
    .execute(&client)
    .await?;

insert_into::<User>()
    .values(User { id: 2, name: "Ana".to_string() })
    .enqueue_in("user.reminder", 30_000)
    .execute(&client)
    .await?;

insert_into::<User>()
    .values(User { id: 3, name: "Lia".to_string() })
    .enqueue_at("user.scheduled", dinoco::Utc::now() + chrono::Duration::hours(1))
    .execute(&client)
    .await?;
```

## Worker

Verwenden Sie `workers()`, um asynchrone Handler zu registrieren. Der Worker-Kontext stellt Folgendes bereit:

- `data`: der neu hydrierte Datensatz.
- `client`: eine Kopie des `DinocoClient`.
- `success()` oder `remove()`: entfernt den Job aus der Warteschlange.
- `fail()`: plant mit dem Standard-Retry neu.
- `retry_in()` und `retry_at()`: gleiche Idee wie `enqueue_in` und `enqueue_at`.
- `.run().await?`: startet den Worker im Hintergrund und kehrt sofort mit einem `JoinHandle` zurück.

Beim Aufruf von `run` erstellt Dinoco einen neuen, exklusiven `DinocoClient` für die Worker mit einem eigenen Verbindungspool. So nutzt der Warteschlangen-Loop nicht denselben Pool wie die Hauptanwendung.

```rust
use database::*;

let _worker = workers()
    .on::<User, _, _>("user.created", |job| async move {
        println!("Verarbeite {}", job.data.name);
        job.success()
    })
    .run()
    .await?;
```

Wenn Sie eine Projektion mit Beziehungen benötigen, verwenden Sie `on_with_relation`. Dinoco lädt den Datensatz neu und wendet den Include-Baum an, bevor der Handler ausgeführt wird.

```rust
use database::*;

let _worker = workers()
    .on_with_relation::<User, UserWithPosts, _, _, _, _>(
        "user.created",
        |user| user.posts().select::<PostListItem>(),
        |job| async move {
            println!("{} hat {} Beiträge", job.data.name, job.data.posts.len());
            job.success()
        },
    )
    .run()
    .await?;
```

Sie können auch mit Listen arbeiten:

```rust
use database::*;

let _worker = workers()
    .on::<Vec<User>, _, _>("user.batch-updated", |job| async move {
        for user in job.data {
            println!("{}", user.name);
        }

        job.success()
    })
    .run()
    .await?;
```

## Redis-Konfiguration

Die Warteschlangen verwenden das in `DinocoClientConfig::with_redis(...)` konfigurierte `Redis`. Für `enqueue_in` und `enqueue_at` ist es ratsam, die Persistenz zu aktivieren, da geplante Jobs bis zur Ausführungszeit in Redis leben.

Empfehlung: Verwenden Sie **AOF + RDB zusammen**.

```conf
# AOF aktivieren
appendonly yes

# fsync-Häufigkeit
appendfsync everysec

# AOF-Datei
appendfilename "appendonly.aof"

# --- SNAPSHOT (RDB) ---
save 900 1
save 300 10
save 60 10000

# --- SICHERHEIT ---
dir /data

# --- PERFORMANCE ---
no-appendfsync-on-rewrite yes
```

## Anmerkungen

- `delete` und `delete_many` unterstützen `enqueue` nicht.
- Der Worker ist vollständig asynchron und verwendet `tokio` unter der Haube.
- `run()` blockiert den Rest der Anwendung nicht: Es startet den Loop im Hintergrund.
- `Option\<User\>` wird bei der Worker-Registrierung akzeptiert, aber der Handler wird nur ausgeführt, wenn ein Datensatz zur Re-Hydrierung vorhanden ist.
