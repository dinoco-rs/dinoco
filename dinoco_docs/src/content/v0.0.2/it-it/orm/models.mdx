# Modelli

I `model` definiscono le entità centrali della tua applicazione nello schema di Dinoco. Ogni `model` normalmente rappresenta una tabella nel database e serve come base per la generazione di codice, query tipizzate e operazioni con l'API di Dinoco.

---

## Cosa rappresenta un modello

Un `model` descrive:

- Il nome dell'entità.
- I campi memorizzati nel database.
- Quali campi sono obbligatori o opzionali.
- Quali campi sono unici o identificatori.
- Come questi dati verranno utilizzati dal codegen e dall'API.

Esempio:

```dinoco
model User {
	id    Integer @id @default(autoincrement())
	email String  @unique
	name  String?
}
```

In questo esempio:

- `User` è il model.
- `id`, `email` e `name` sono campi scalari.
- `id` è l'identificatore principale.
- `email` ha una restrizione di unicità.

## Esempio completo

Uno schema semplice con un modello di solito si presenta così:

```dinoco
config {
	database = "postgresql"
	database_url = env("DATABASE_URL")
}

model User {
	id        Integer  @id @default(autoincrement())
	email     String   @unique
	name      String?
	active    Boolean  @default(true)
	createdAt DateTime @default(now())
}

model Post {
	id        Integer  @id @default(autoincrement())
	title     String
	content   String?
	published Boolean  @default(false)
	createdAt DateTime @default(now())
}
```

## Struttura di un campo

Ogni campo di un model è composto da:

- Nome
- Tipo
- Modificatore opzionale
- Attributi opzionali

Esempio:

```dinoco
email String @unique
```

In questa riga:

- `email` è il nome del campo.
- `String` è il tipo.
- `@unique` è un attributo.

## Tipi di campo

I campi possono rappresentare valori di schema di base, come testo, numero, booleano e date.

### Campi scalari

Sono campi che memorizzano valori diretti, come testo, numero, booleano e date.

```dinoco
model Product {
	id          Integer  @id @default(autoincrement())
	name        String
	description String?
	price       Float
	active      Boolean  @default(true)
	createdAt   DateTime @default(now())
}
```

## Modificatori di tipo

Dinoco supporta due modificatori principali:

| Modificatore | Significato    | Esempio         |
| :---------- | :------------- | :-------------- |
| `?`         | Campo opzionale | `name String?`  |
| `[]`        | Lista          | `tags String[]` |

### Campo opzionale

```dinoco
model User {
	id   Integer @id @default(autoincrement())
	name String?
}
```

`name` può essere nullo o assente, a seconda del database e del livello generato.

### Campo in lista

```dinoco
model Article {
	id   Integer  @id @default(autoincrement())
	tags String[]
}
```

Questo formato rappresenta un elenco di valori quando il database e il flusso supportano questo tipo di struttura.

## Attributi più comuni

Gli attributi modificano il comportamento dei campi e del modello.

| Attributo        | Uso                               |
| :-------------- | :-------------------------------- |
| `@id`           | Definisce l'identificatore principale |
| `@default(...)` | Definisce un valore predefinito   |
| `@unique`       | Garantisce l'unicità              |

### `@id`

Definisce il campo che identifica univocamente un record.

```dinoco
id Integer @id @default(autoincrement())
```

Ogni model deve avere un identificatore chiaro affinché l'API generata possa operare in sicurezza.

### `@default(...)`

Definisce un valore predefinito per il campo.

```dinoco
active    Boolean  @default(true)
createdAt DateTime @default(now())
id        Integer  @default(autoincrement())
```

Funzioni e valori comuni:

| Esempio                     | Uso                 |
| :-------------------------- | :------------------ |
| `@default(false)`           | Booleano predefinito |
| `@default(now())`           | Data attuale        |
| `@default(autoincrement())` | Intero incrementale |
| `@default(uuid())`          | Identificatore UUID |

### `@unique`

Garantisce che il valore del campo non si ripeta.

```dinoco
model User {
	id    Integer @id @default(autoincrement())
	email String  @unique
}
```

Questo attributo è ideale per campi come email, username e codici esterni.

## Decoratori di model

Oltre agli attributi nei singoli campi, Dinoco supporta anche i decoratori applicati all'intero blocco del model.

| Decoratore            | Uso                                   |
| :-------------------- | :------------------------------------ |
| `@@ids([...])`        | Definisce una chiave primaria composta |
| `@@table_name("...")` | Mappa il nome reale della tabella nel database |

### `@@ids([...])`

Usa `@@ids` quando l'identità del record dipende da più di un campo.

```dinoco
model Membership {
	userId Integer
	teamId Integer
	role   String

	@@ids([userId, teamId])
}
```

Questo formato è utile nelle tabelle associative e negli scenari in cui l'unicità naturale è già composta.

### `@@table_name("...")`

Usa `@@table_name()` quando vuoi mantenere un nome di model più amichevole nello schema, ma mapparlo a un altro nome fisico nel database.

```dinoco
model User {
	id    Integer @id @default(autoincrement())
	email String  @unique

	@@table_name("users")
}
```

In questo caso:

- Il model continua a chiamarsi `User` nello schema e nell'API generata.
- La tabella fisica nel database diventa `users`.

## Esempio di model utente

```dinoco
model User {
	id        Integer  @id @default(autoincrement())
	email     String   @unique
	name      String?
	active    Boolean  @default(true)
	createdAt DateTime @default(now())
}
```

Dopo il codegen, questo model può essere utilizzato direttamente con l'API di Dinoco.

## Esempio di ricerca di utenti con l'API di Dinoco

### Cercare un singolo record

```rust
let user = dinoco::find_first::<User>()
    .cond(|x| x.id.eq(1_i64))
    .execute(&client)
    .await?;
```

### Cercare più record

```rust
let users = dinoco::find_many::<User>()
    .cond(|x| x.name.includes("Ana"))
    .order_by(|x| x.id.asc())
    .take(10)
    .execute(&client)
    .await?;
```

## Esempio di creazione di un utente con l'API di Dinoco

```rust
dinoco::insert_into::<User>()
    .values(User {
        id: 0,
        email: "bia@dinoco.rs".to_string(),
        name: Some("Bia".to_string()),
        active: true,
        createdAt: dinoco::Utc::now(),
    })
    .execute(&client)
    .await?;
```

## Esempio di aggiornamento di un utente con l'API di Dinoco

```rust
dinoco::update::<User>()
    .cond(|x| x.id.eq(1_i64))
    .values(User {
        id: 1,
        email: "bia@dinoco.rs".to_string(),
        name: Some("Beatriz".to_string()),
        active: true,
        createdAt: dinoco::Utc::now(),
    })
    .execute(&client)
    .await?;
```

Se desideri aggiornamenti atomici su un singolo campo, il flusso `find_and_update` è solitamente ancora più diretto:

```rust
let user = dinoco::find_and_update::<User>()
    .cond(|x| x.id.eq(1_i64))
    .update(|x| x.name.set("Beatriz"))
    .execute(&client)
    .await?;
```

## Esempio di rimozione di un utente con l'API di Dinoco

```rust
dinoco::delete::<User>()
    .cond(|x| x.id.eq(1_i64))
    .execute(&client)
    .await?;
```

Per rimozioni in blocco:

```rust
dinoco::delete_many::<User>()
    .cond(|x| x.active.eq(false))
    .execute(&client)
    .await?;
```

## Riepilogo rapido

| Concetto       | Esempio                | Obiettivo                   |
| :------------- | :--------------------- | :-------------------------- |
| Model          | `model User { ... }`   | Rappresentare un'entità    |
| Campo scalare  | `email String`         | Memorizzare un valore semplice |
| Campo opzionale | `name String?`         | Permettere l'assenza di valore |
| Campo in lista | `tags String[]`        | Memorizzare più valori      |
| ID             | `id Integer @id`       | Identificare univocamente   |
| Default        | `@default(now())`      | Compilare automaticamente   |
| Unique         | `email String @unique` | Evitare duplicati           |

## Quando creare un nuovo model

Normalmente crei un nuovo `model` quando un'entità della tua applicazione necessita di:

- Essere persistita nel database.
- Avere una propria identità.
- Essere interrogata isolatamente.
- Avere proprie regole di lettura e scrittura.

Esempi comuni:

- `User`
- `Post`
- `Comment`
- `Category`
- `Order`
- `Invoice`
