# キュー

Dinocoのキューシステムは、`insert`、`find`、`update`から非同期イベントを公開することを可能にし、ワーカーが実行する際にレコードを再ハイドレートするために必要なデータのみを`Redis`に保存します。

## できること

- 削除操作を除くすべてのクライアントメソッドで、.enqueue(...)、.enqueue_in(...)、.enqueue_at(...) のチェーンを許可します。
- `workers().on::<Model>(... )` を使用してジョブを処理します。
- `workers().on_with_relation::<Model, Projection>(... )` を使用して、ワーカーでリレーションを持つプロジェクションをロードします。
- ハンドラーが実行される前に最新のデータを再ロードします。
- レコードがもはや存在しないジョブを自動的に無視します。

## 仕組み

`enqueue`を使用すると、Dinocoは次のことを行います。

1. メイン操作を通常通り実行します。
2. レコードの検索条件のみをキューに保存します。
3. ワーカーで、データベースから最新のデータを取得します。
4. その後、ハンドラーを実行します。

検索時にレコードがもはや存在しない場合、ハンドラーは実行されず、ジョブはキューから削除されます。

## キューイングメソッド

- `.enqueue(evento)`: 即時実行のためにスケジュールします。
- `.enqueue_in(evento, delay_ms)`: ミリ秒単位の遅延でスケジュールします。
- `.enqueue_at(evento, data_utc)`: `chrono::DateTime<Utc>`で特定の時刻にスケジュールします。

```rust
use database::*;

insert_into::<User>()
    .values(User { id: 1, name: "Matheus".to_string() })
    .enqueue("user.created")
    .execute(&client)
    .await?;

insert_into::<User>()
    .values(User { id: 2, name: "Ana".to_string() })
    .enqueue_in("user.reminder", 30_000)
    .execute(&client)
    .await?;

insert_into::<User>()
    .values(User { id: 3, name: "Lia".to_string() })
    .enqueue_at("user.scheduled", dinoco::Utc::now() + chrono::Duration::hours(1))
    .execute(&client)
    .await?;
```

## ワーカー

`workers()`を使用して非同期ハンドラーを登録します。ワーカーのコンテキストは以下を公開します。

- `data`: 再ハイドレートされたレコード。
- `client`: `DinocoClient`のコピー。
- `success()` または `remove()`: キューからジョブを削除します。
- `fail()`: 標準のリトライで再スケジュールします。
- `retry_in()` および `retry_at()`: `enqueue_in`および`enqueue_at`と同じ考え方です。
- `.run().await?`: ワーカーをバックグラウンドで起動し、`JoinHandle`をすぐに返します。

`run`を呼び出すと、Dinocoはワーカー専用の新しい`DinocoClient`を作成し、独自の接続プールを持ちます。これにより、キューのループがメインアプリケーションと同じプールを再利用することはありません。

```rust
use database::*;

let _worker = workers()
    .on::<User, _, _>("user.created", |job| async move {
        println!("{}を処理中", job.data.name);
        job.success()
    })
    .run()
    .await?;
```

リレーションを持つプロジェクションが必要な場合は、`on_with_relation`を使用します。Dinocoはハンドラーが実行される前にレコードを再ロードし、インクルードツリーを適用します。

```rust
use database::*;

let _worker = workers()
    .on_with_relation::<User, UserWithPosts, _, _, _, _>(
        "user.created",
        |user| user.posts().select::<PostListItem>(),
        |job| async move {
            println!("{}は{}件の投稿を持っています", job.data.name, job.data.posts.len());
            job.success()
        },
    )
    .run()
    .await?;
```

リストを扱うこともできます。

```rust
use database::*;

let _worker = workers()
    .on::<Vec<User>, _, _>("user.batch-updated", |job| async move {
        for user in job.data {
            println!("{}", user.name);
        }

        job.success()
    })
    .run()
    .await?;
```

## Redisの設定

キューは`DinocoClientConfig::with_redis(...)`で設定された`Redis`を使用します。`enqueue_in`と`enqueue_at`の場合、スケジュールされたジョブは実行時刻までRedisに存在するため、永続性を有効にすることをお勧めします。

推奨: **AOF + RDB を一緒に**使用してください。

```conf
# AOFを有効にする
appendonly yes

# fsyncの頻度
appendfsync everysec

# AOFファイル
appendfilename "appendonly.aof"

# --- スナップショット (RDB) ---
save 900 1
save 300 10
save 60 10000

# --- セキュリティ ---
dir /data

# --- パフォーマンス ---
no-appendfsync-on-rewrite yes
```

## 注意事項

- `delete`および`delete_many`は`enqueue`をサポートしていません。
- ワーカーは完全に非同期であり、内部で`tokio`を使用しています。
- `run()`はアプリケーションの残りの部分をブロックしません。バックグラウンドでループを起動します。
- `Option<User>`はワーカーの登録で受け入れられますが、ハンドラーは再ハイドレートするレコードがある場合にのみ実行されます。
