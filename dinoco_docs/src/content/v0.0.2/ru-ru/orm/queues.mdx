# queues

Система очередей Dinoco позволяет публиковать асинхронные события из `insert`, `find` и `update`, сохраняя в `Redis` только данные, необходимые для восстановления записи, когда воркер будет выполнять задание.

## Что вы можете сделать

- Разрешить цепочку вызовов .enqueue(...), .enqueue_in(...) и .enqueue_at(...) для всех методов клиента, за исключением операций удаления.
- Обрабатывать задания с помощью `workers().on::\<Model\>(... )`.
- Загружать проекции с отношениями в воркере, используя `workers().on_with_relation::\<Model, Projection\>(... )`.
- Перезагружать самые свежие данные перед запуском обработчика.
- Автоматически игнорировать задания, запись которых больше не существует.

## Как это работает

Когда вы используете `enqueue`, Dinoco:

1. Выполняет основную операцию в обычном режиме.
2. Сохраняет в очередь только критерии поиска записи.
3. В воркере извлекает самые свежие данные из базы данных.
4. Только после этого выполняет обработчик.

Если запись больше не существует на момент поиска, обработчик не выполняется, и задание удаляется из очереди.

## Методы постановки в очередь

- `.enqueue(evento)`: планирует для немедленного выполнения.
- `.enqueue_in(evento, delay_ms)`: планирует с задержкой в миллисекундах.
- `.enqueue_at(evento, data_utc)`: планирует на определенную дату в `chrono::DateTime\<Utc\>`.

```rust
use database::*;

insert_into::<User>()
    .values(User { id: 1, name: "Matheus".to_string() })
    .enqueue("user.created")
    .execute(&client)
    .await?;

insert_into::<User>()
    .values(User { id: 2, name: "Ana".to_string() })
    .enqueue_in("user.reminder", 30_000)
    .execute(&client)
    .await?;

insert_into::<User>()
    .values(User { id: 3, name: "Lia".to_string() })
    .enqueue_at("user.scheduled", dinoco::Utc::now() + chrono::Duration::hours(1))
    .execute(&client)
    .await?;
```

## Воркер

Используйте `workers()` для регистрации асинхронных обработчиков. Контекст воркера предоставляет:

- `data`: восстановленная запись.
- `client`: копия `DinocoClient`.
- `success()` или `remove()`: удаляет задание из очереди.
- `fail()`: перепланирует со стандартной политикой повторных попыток.
- `retry_in()` и `retry_at()`: та же идея, что и у `enqueue_in` и `enqueue_at`.
- `.run().await?`: запускает воркер в фоновом режиме и немедленно возвращает `JoinHandle`.

При вызове `run` Dinoco создает новый `DinocoClient` исключительно для воркеров, с собственным пулом соединений. Таким образом, цикл очереди не использует повторно тот же пул, что и основное приложение.

```rust
use database::*;

let _worker = workers()
    .on::<User, _, _>("user.created", |job| async move {
        println!("Обработка {}", job.data.name); // Обработка
        job.success()
    })
    .run()
    .await?;
```

Если вам нужна проекция с отношениями, используйте `on_with_relation`. Dinoco перезагружает запись и применяет дерево включений перед запуском обработчика.

```rust
use database::*;

let _worker = workers()
    .on_with_relation::<User, UserWithPosts, _, _, _, _>(
        "user.created",
        |user| user.posts().select::<PostListItem>(),
        |job| async move {
            println!("{} имеет {} постов", job.data.name, job.data.posts.len()); // имеет постов
            job.success()
        },
    )
    .run()
    .await?;
```

Вы также можете работать со списками:

```rust
use database::*;

let _worker = workers()
    .on::<Vec<User>, _, _>("user.batch-updated", |job| async move {
        for user in job.data {
            println!("{}", user.name);
        }

        job.success()
    })
    .run()
    .await?;
```

## Конфигурация Redis

Очереди используют `Redis`, настроенный в `DinocoClientConfig::with_redis(...)`. Для `enqueue_in` и `enqueue_at` рекомендуется включить персистентность, потому что запланированные задания хранятся в Redis до времени выполнения.

Рекомендация: используйте **AOF + RDB вместе**.

```conf
# Активировать AOF
appendonly yes

# Частота fsync
appendfsync everysec

# Файл AOF
appendfilename "appendonly.aof"

# --- СНИМОК (RDB) ---
save 900 1
save 300 10
save 60 10000

# --- БЕЗОПАСНОСТЬ ---
dir /data

# --- ПРОИЗВОДИТЕЛЬНОСТЬ ---
no-appendfsync-on-rewrite yes
```

## Примечания

- `delete` и `delete_many` не поддерживают `enqueue`.
- Воркер полностью асинхронен и использует `tokio` в фоновом режиме.
- `run()` не блокирует остальную часть приложения: он запускает цикл в фоновом режиме.
- `Option\<User\>` принимается при регистрации воркера, но обработчик запускается только при наличии записи для восстановления.
