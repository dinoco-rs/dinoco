# queues

Система очередей Dinoco позволяет публиковать асинхронные события из `insert`, `find` и `update`, сохраняя в `Redis` только необходимые данные для повторной гидратации записи при выполнении воркером.

## Что вы можете делать

- Разрешить цепочку .enqueue(...), .enqueue_in(...) и .enqueue_at(...) во всех методах клиента, кроме операций удаления.
- Обрабатывать задачи с помощью `workers().on::<Model>(... )`.
- Перезагружать самые свежие данные перед запуском обработчика.
- Автоматически игнорировать задачи, записи которых больше не существуют.

## Как это работает

Когда вы используете `enqueue`, Dinoco:

1. Выполняет основную операцию в обычном режиме.
2. Сохраняет в очередь только критерии поиска записи.
3. В воркере извлекает самые свежие данные из базы данных.
4. Только затем выполняет обработчик.

Если запись больше не существует на момент поиска, обработчик не запускается, и задача удаляется из очереди.

## Методы постановки в очередь

- `.enqueue(событие)`: планирует немедленное выполнение.
- `.enqueue_in(событие, delay_ms)`: планирует с задержкой в миллисекундах.
- `.enqueue_at(событие, data_utc)`: планирует на определенную дату в `chrono::DateTime<Utc>`.

```rust
use database::*;

insert_into::<User>()
    .values(User { id: 1, name: "Matheus".to_string() })
    .enqueue("user.created")
    .execute(&client)
    .await?;

insert_into::<User>()
    .values(User { id: 2, name: "Ana".to_string() })
    .enqueue_in("user.reminder", 30_000)
    .execute(&client)
    .await?;

insert_into::<User>()
    .values(User { id: 3, name: "Lia".to_string() })
    .enqueue_at("user.scheduled", dinoco::Utc::now() + chrono::Duration::hours(1))
    .execute(&client)
    .await?;
```

## Воркер

Используйте `workers()` для регистрации асинхронных обработчиков. Контекст воркера предоставляет:

- `data`: повторно гидратированная запись.
- `client`: копия `DinocoClient`.
- `success()` или `remove()`: удаляет задачу из очереди.
- `fail()`: перепланирует с использованием стандартной политики повторных попыток.
- `retry_in()` и `retry_at()`: та же идея, что и `enqueue_in` и `enqueue_at`.
- `.run().await?`: запускает воркер в фоновом режиме и немедленно возвращает `JoinHandle`.

При вызове `run` Dinoco создает новый `DinocoClient` исключительно для воркеров, с собственным пулом соединений. Таким образом, цикл очереди не использует тот же пул, что и основное приложение.

```rust
use database::*;

let _worker = workers()
    .on::<User, _, _>("user.created", |job| async move {
        println!("Processando {}", job.data.name); // Обработка {}
        job.success()
    })
    .run()
    .await?;
```

Вы также можете работать со списками:

```rust
use database::*;

let _worker = workers()
    .on::<Vec<User>, _, _>("user.batch-updated", |job| async move {
        for user in job.data {
            println!("{}", user.name);
        }

        job.success()
    })
    .run()
    .await?;
```

## Конфигурация Redis

Очереди используют `Redis`, настроенный в `DinocoClientConfig::with_redis(...)`. Для `enqueue_in` и `enqueue_at` рекомендуется включить персистентность, потому что запланированные задачи живут в Redis до времени выполнения.

Рекомендация: используйте **AOF + RDB вместе**.

```conf
# Активировать AOF
appendonly yes

# Частота fsync
appendfsync everysec

# Файл AOF
appendfilename "appendonly.aof"

# --- СНИМОК (RDB) ---
save 900 1
save 300 10
save 60 10000

# --- БЕЗОПАСНОСТЬ ---
dir /data

# --- ПРОИЗВОДИТЕЛЬНОСТЬ ---
no-appendfsync-on-rewrite yes
```

## Примечания

- `delete` и `delete_many` не поддерживают `enqueue`.
- Воркер полностью асинхронен и использует `tokio` под капотом.
- `run()` не блокирует остальную часть приложения: он запускает цикл в фоновом режиме.
- `Option<User>` принимается при регистрации воркера, но обработчик запускается только при наличии записи для повторной гидратации.
